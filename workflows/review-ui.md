# Review UI Workflow

For any change that touches the visual surface, tenant content, component
public APIs, or `features/` composition.

This workflow is **read-only first**. Do NOT modify files initially.

---

## Step 0 — Read first

```
FRAMIXOR.md
.planning/CURRENT_STATE.md
.planning/OPEN_BLOCKERS.md
rules/frontend-architecture.md
rules/ddd-boundaries.md
rules/visual-contracts.md
rules/testing.md
rules/lovable-rules.md
docs/contracts/*.md
```

Identify which contracts the change touches. If a touched contract has no
contract document, that itself is a finding.

---

## Step 1 — Inventory the change

Without modifying anything, enumerate:

1. Files added.
2. Files modified.
3. Files deleted.
4. Layers touched (`app/features/shared/tenants/domain/testing`).
5. Contracts touched (visual tokens, tenant content, component APIs).
6. Tests added / modified / deleted.
7. Dependencies added / modified.

If files outside the layer scope of the stated goal were modified, that is a
finding.

---

## Step 2 — Return findings

Return, in writing, a structured review with these sections.

### A. Contract violations

* Token names added/removed/renamed without ADR.
* Component public-API changes that are breaking.
* Tenant content shape changes that are breaking.
* Inline visual values (hex, px, ms) introduced.
* Hardcoded tenant copy introduced outside `tenants/`.

### B. Boundary violations

* Forbidden import edges (`features` -> `app`, `shared` -> `features`,
  `domain` -> anything, etc.).
* `domain/` importing React / DOM / fetch / storage.
* Production code importing from `testing/`.
* Cross-feature imports.

### C. Test violations

* Missing tests for new public surface.
* Snapshot-only tests where behavior tests are required.
* Mocks of children or React internals.
* `expect(true).toBe(true)`-style placeholder assertions.
* Tests asserting token *values* (only existence is allowed).
* Tests asserting tenant *copy* (only shape is allowed).

### D. Lovable rule violations (if Lovable authored the change)

* Any modification to `domain/`, `tenants/`, `docs/contracts/`, `rules/`,
  `workflows/`, `*.test.ts(x)`, `src/testing/`, `package.json`, lockfiles, or
  CI config.
* Token *name* changes.
* Component public-API changes.
* Adapter changes (new direct `fetch` / storage / `window` calls).

### E. SOLID notes

* Boolean prop explosions.
* God-prop types.
* Domain logic leaking into components.
* Concrete adapters imported by features instead of typed abstractions.

### F. Accessibility checklist (when UI is rendered)

* Interactive elements have a role and an accessible name.
* Focus order is sensible.
* Color contrast is not the *only* signal of state.
* Decorative images have empty `alt`; meaningful images have descriptive `alt`.
* Tests query by role / label / text rather than class / `data-testid` when
  possible.

### G. Tenant assumptions

* No assumption that a specific tenant always exists.
* No assumption that a specific tenant field is always present beyond what the
  contract guarantees.
* No conditional logic keyed on a tenant id literal.

### H. CI status

* `npx tsc --noEmit` — green / red.
* `npm test -- --run` — green / red.
* `npm run build` — green / red.

### I. Minimal patch plan

If findings exist, propose the smallest patch that resolves them. Do not
propose unrelated improvements.

### J. Verdict

One of:

* `APPROVE` — no findings or only non-blocking notes.
* `APPROVE WITH NITS` — non-blocking suggestions, change can merge.
* `REQUEST CHANGES` — at least one blocking finding (any item in A, B, C, D
  is blocking by default).
* `REJECT` — change should not exist in its current form (e.g., touches
  rules/workflows/contracts without an ADR, introduces a forbidden
  dependency).

---

## Step 3 — Only after approval

Generate any patch only after the verdict is `REQUEST CHANGES` or
`APPROVE WITH NITS` and the author has accepted the patch plan.

Patches generated by the reviewer follow `workflows/feature-implementation.md`
from Step 1 onward (their own plan, their own tests, their own report entry).

---

## Forbidden during review

* Modifying files before Step 1 is complete.
* "Approve to unblock" without re-reading the contracts.
* Treating Lovable-authored changes more leniently because they are
  Lovable-authored.
* Ignoring CI red.
